# SPA Salon API — лабораторная работа по веб-программированию

## Описание проекта

Данный проект представляет собой REST API для SPA-салона, разработанный на FastAPI.

Проект предназначен для демонстрации базовых принципов работы backend-приложения:

* запуск веб-сервера;
* работа с REST API;
* автоматическая документация Swagger;
* регистрация пользователя;
* вход пользователя;
* выход пользователя;
* получение текущего пользователя;
* просмотр списка пользователей;
* передача данных в формате JSON.

API тестируется через Swagger UI, который автоматически создаётся FastAPI.

---

## Используемые технологии

В проекте используются:

* Python
* FastAPI
* Uvicorn
* Swagger / OpenAPI
* Pydantic
* JSON
* HTTP-запросы GET и POST

---

## Назначение лабораторной работы

Цель лабораторной работы — изучить создание REST API на FastAPI и проверить работу основных API-эндпоинтов через Swagger.

В рамках работы реализованы следующие возможности:

1. Проверка работоспособности сервера.
2. Регистрация пользователя.
3. Авторизация пользователя.
4. Получение информации о текущем пользователе.
5. Получение списка пользователей.
6. Выход из системы.

## Установка и запуск проекта

### 1. Распаковка проекта

Распаковать архив с проектом и перейти в папку проекта:

```bash
cd lab3
```

---

### 2. Создание виртуального окружения

Для Windows:

```bash
python -m venv venv
```

Активация виртуального окружения:

```bash
venv\Scripts\activate
```

После активации в терминале должно появиться:

```text
(venv)
```

---

### 3. Установка зависимостей

Перед установкой зависимостей рекомендуется обновить `pip`, `setuptools` и `wheel`:

```bash
python.exe -m pip install --upgrade pip setuptools wheel
```

Затем установить зависимости:

```bash
pip install -r requirements.txt
```

Рабочий вариант `requirements.txt` для текущей демонстрационной версии проекта:

```txt
fastapi==0.104.1
uvicorn[standard]==0.24.0
sqlalchemy==2.0.23
alembic==1.12.1
python-dotenv==1.0.0
pydantic-settings==2.1.0
email-validator
```

---

### 4. Запуск сервера

Запуск FastAPI-приложения:

```bash
uvicorn app.main:app --reload --host 127.0.0.1 --port 8000
```

После запуска сервер будет доступен по адресу:

```text
http://127.0.0.1:8000
```

Swagger-документация доступна по адресу:

```text
http://127.0.0.1:8000/api/docs
```

## Проверка работы API

После запуска проекта нужно открыть Swagger:

```text
http://127.0.0.1:8000/api/docs
```

На странице Swagger будут доступны все эндпоинты API.

---

## Основные эндпоинты

### 1. Проверка сервера

```http
GET /health
```

Назначение:

Проверяет, работает ли сервер.

Пример ответа:

```json
{
  "status": "ok"
}
```

---

### 2. Регистрация пользователя

```http
POST /api/v1/auth/register
```

Назначение:

Создаёт нового пользователя.

Пример JSON-запроса:

```json
{
  "email": "user@example.com",
  "password": "password123",
  "full_name": "Иван Петров"
}
```

Пример ответа:

```json
{
  "message": "User user@example.com registered successfully"
}
```

---

### 3. Вход пользователя

```http
POST /api/v1/auth/login
```

Назначение:

Проверяет email и пароль пользователя.

Пример JSON-запроса:

```json
{
  "email": "user@example.com",
  "password": "password123"
}
```

Пример ответа:

```json
{
  "message": "Login successful"
}
```

---

### 4. Получение текущего пользователя

```http
GET /api/v1/auth/me
```

Назначение:

Возвращает информацию о текущем пользователе.

В текущей демонстрационной версии метод возвращает тестовое сообщение.

Пример ответа:

```json
{
  "message": "Current user info - implement with real token"
}
```

---

### 5. Получение списка пользователей

```http
GET /api/v1/users/
```

Назначение:

Возвращает список зарегистрированных пользователей.

Пример ответа:

```json
[
  {
    "email": "user@example.com",
    "password": "password123",
    "full_name": "Иван Петров"
  }
]
```

Важно: в учебной демонстрационной версии пароль может отображаться в ответе API. В реальном приложении пароль не должен возвращаться клиенту и должен храниться только в виде хеша.

---

### 6. Выход пользователя

```http
POST /api/v1/auth/logout
```

Назначение:

Завершает пользовательскую сессию.

Пример ответа:

```json
{
  "message": "Logout successful"
}
```

## Что такое REST API

REST API — это способ взаимодействия клиента и сервера через HTTP-запросы.

В проекте используются основные HTTP-методы:

```text
GET  — получение данных
POST — отправка данных на сервер
```

Примеры:

```http
GET /health
POST /api/v1/auth/register
POST /api/v1/auth/login
GET /api/v1/users/
POST /api/v1/auth/logout
```

---

## Что такое Swagger

Swagger — это инструмент для автоматической документации и тестирования API.

В данном проекте Swagger позволяет:

* увидеть все доступные эндпоинты;
* посмотреть формат запросов;
* отправить тестовые данные;
* получить ответ от сервера;
* проверить работу API без отдельного frontend-приложения.

Swagger открывается по адресу:

```text
http://127.0.0.1:8000/api/docs
```

---

## Что такое JSON

JSON — это формат передачи данных между клиентом и сервером.

Пример JSON:

```json
{
  "email": "user@example.com",
  "password": "password123"
}
```

В данной лабораторной JSON используется при регистрации и входе пользователя.

## Особенности текущей версии

Текущая версия проекта является демонстрационной.

Особенности:

* пользователи хранятся временно в памяти приложения;
* после перезапуска сервера список пользователей очищается;
* PostgreSQL подготовлен в структуре проекта, но фактически не используется в текущей версии;
* авторизация реализована в упрощённом виде;
* пароль в учебной версии может отображаться в ответе API.

В реальном приложении необходимо:

* хранить пользователей в базе данных;
* хешировать пароли;
* не возвращать пароль в ответах API;
* реализовать полноценную JWT-авторизацию;
* разграничить доступ к защищённым эндпоинтам.

## Возможные ошибки

### Ошибка с `psycopg2-binary`

Если при установке зависимостей появляется ошибка:

```text
Error: pg_config executable not found
```

или:

```text
Failed to build psycopg2-binary
```

Это связано с драйвером PostgreSQL.

Решение для текущей лабораторной:

1. Открыть `requirements.txt`.
2. Удалить строку:

```txt
psycopg2-binary
```

или:

```txt
psycopg2
```

3. Повторить установку:

```bash
pip install -r requirements.txt
```

Так как текущая демонстрационная версия не использует PostgreSQL напрямую, проект будет работать без этого пакета.
